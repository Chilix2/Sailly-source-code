# Before & After Comparison

## Latency Measurements

### BEFORE

| Metric | Value | What It Measures | Issue |
|--------|-------|------------------|-------|
| `llm_latency_ms` | 2388ms | STT final → first TTS text | Brain processing only |
| `tts_first_byte_ms` | NULL | Would be TTS synthesis | Never populated, `_turn_timings` not initialized |
| User Perceived Latency | ??? | Unknown | Not measured at all |

**Problem**: We were measuring only brain latency (2388ms), completely missing TTS synthesis time (~800ms) and network delay, so the actual user-perceived latency was unknown.

---

### AFTER

| Metric | Value | What It Measures | Status |
|--------|-------|------------------|--------|
| `brain_processing_ms` (llm_latency_ms) | 2388ms | STT final → first TTS text | ✅ Still captured |
| `tts_first_byte_ms` | 800ms | l2_done → first audio byte | ✅ Now populated (synthesis + network) |
| `tts_ttfb_ms` | 3188ms | STT final → first audio byte | ✅ NEW: End-to-end TTFB |
| User Perceived Latency | 3188ms | Total latency felt by user | ✅ Now measured! |

**Result**: We now capture the complete picture:
- **Brain processing**: 2388ms
- **TTS synthesis + network**: 800ms
- **Total perceived**: 3188ms = 2388ms + 800ms

```markdown
## Latency Breakdown

- **Brain Processing** (`llm_latency_ms`): Time from STT final to first TTS text generated by LLM
- **TTS TTFB** (`tts_ttfb_ms`): End-to-end time from STT final to first audio byte sent to client
  - Includes: LLM processing + TTS synthesis + network delay
  - Perceived latency by user ≈ TTS TTFB
- **TTS Synthesis** = TTS TTFB - Brain Processing (synthesis + network)
```

✅ **Improvements**:
- Shows both brain processing and end-to-end TTFB
- Explains what each metric represents
- Clarifies that `tts_ttfb_ms` (~3188ms) ≈ perceived latency
- Helps identify bottlenecks (if 3188ms > 2388ms, TTS/network is slow)

## Database Records

### BEFORE

```sql
SELECT 
    turn_number,
    llm_latency_ms,
    tts_first_byte_ms,
    tts_ttfb_ms,
    total_latency_ms
FROM google_turn_metrics
WHERE call_sid = 'demo-xyz'
ORDER BY turn_number;
```

**Output**:
```
turn_number | llm_latency_ms | tts_first_byte_ms | tts_ttfb_ms | total_latency_ms
1           | 2388           | NULL              | NULL        | 3200
2           | 2450           | NULL              | NULL        | 3220
3           | 2400           | NULL              | NULL        | 3240
```

❌ **Problems**:
- `tts_first_byte_ms` is always NULL (never initialized)
- `tts_ttfb_ms` doesn't exist (column not in schema)
- Can't measure actual TTS performance
- Can't identify slow TTS synthesizer vs slow network

---

### AFTER

```sql
SELECT 
    turn_number,
    llm_latency_ms as brain_processing_ms,
    tts_first_byte_ms,
    tts_ttfb_ms,
    total_latency_ms,
    (tts_ttfb_ms - llm_latency_ms) as tts_synthesis_network_ms
FROM google_turn_metrics
WHERE call_sid = 'demo-xyz'
ORDER BY turn_number;
```

**Output**:
```
turn_number | brain_processing_ms | tts_first_byte_ms | tts_ttfb_ms | total_latency_ms | tts_synthesis_network_ms
1           | 2388                | 812               | 3200        | 3200             | 812
2           | 2450                | 770               | 3220        | 3220             | 770
3           | 2400                | 840               | 3240        | 3240             | 840
```

✅ **Benefits**:
- All metrics populated with real values
- Can now compute derived metrics: `tts_synthesis_network_ms = tts_ttfb_ms - brain_processing_ms`
- Can identify TTS synthesizer issues (if values suddenly spike)
- Can measure network quality (if tts_synthesis_network_ms is high)
- Can track performance over time

## Latency Analysis

### Example: Identifying a Slow Turn

**Query**: Which turns had high TTS latency?

```sql
SELECT 
    turn_number,
    llm_latency_ms,
    tts_first_byte_ms,
    (tts_ttfb_ms - llm_latency_ms) as tts_latency_total
FROM google_turn_metrics
WHERE call_sid = 'demo-xyz'
ORDER BY (tts_ttfb_ms - llm_latency_ms) DESC
LIMIT 5;
```

**BEFORE**: Impossible - data not available ❌

**AFTER**:
```
turn_number | llm_latency_ms | tts_first_byte_ms | tts_latency_total
5           | 2500           | 1200              | 1200           ← slow TTS!
2           | 2450           | 770               | 770
1           | 2388           | 812               | 812
3           | 2400           | 840               | 840
4           | 2350           | 750               | 750
```

**Insights**:
- Turn 5 had 1200ms TTS latency (vs avg ~800ms) → investigate TTS service
- Could be network issue, TTS synthesizer bottleneck, or high text complexity

## Monitoring Dashboard Ready

### New Metrics Available for Monitoring

```python
# Calculate percentiles across all turns
SELECT 
    call_sid,
    COUNT(*) as turns,
    PERCENTILE_CONT(0.50) WITHIN GROUP (ORDER BY brain_processing_ms) as p50_brain,
    PERCENTILE_CONT(0.95) WITHIN GROUP (ORDER BY brain_processing_ms) as p95_brain,
    PERCENTILE_CONT(0.50) WITHIN GROUP (ORDER BY tts_ttfb_ms) as p50_ttfb,
    PERCENTILE_CONT(0.95) WITHIN GROUP (ORDER BY tts_ttfb_ms) as p95_ttfb,
    PERCENTILE_CONT(0.50) WITHIN GROUP (
        ORDER BY (tts_ttfb_ms - brain_processing_ms)
    ) as p50_tts_synthesis,
    PERCENTILE_CONT(0.95) WITHIN GROUP (
        ORDER BY (tts_ttfb_ms - brain_processing_ms)
    ) as p95_tts_synthesis
FROM google_turn_metrics
WHERE created_at > NOW() - INTERVAL '1 day'
GROUP BY call_sid;
```

**NOW POSSIBLE**:
- Detect slow TTS service (high p95_tts_synthesis)
- Separate brain processing issues from TTS issues
- SLA monitoring: "Is user-perceived latency < 4000ms?"
- Alert when TTS latency > 1500ms (might indicate synthesizer degradation)
