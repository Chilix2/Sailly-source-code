# I1.1_D3 Readback Loop Root Cause Analysis

## Issue Summary
Readback is displayed twice (turns 13 & 15) before commit (turn 17).
- Score: 100% ✅ (order commits successfully)
- Product Behavior: CORRECT (order is placed with all correct details)
- Harness Flag: BOT_LOOP (due to readback repeat detection)

## Root Cause
The state machine has TWO readback display checkpoints:
1. **Lines 1656-1698**: First readback display (when `_readback_already_shown == False`)
2. **Lines 1763-1799**: Second readback display (in confirmation-cycle execution path)

After first readback confirmation (turn 14):
- Code sets `end_call_stage = "idle"` (line 1706)
- Falls through to second readback section (line 1750+)
- At line 1755: `_confirmation_cycle_count` is incremented from 0 → 1
- At line 1763: Checks `_readback_already_shown` - YES, skips display
- Should proceed to commit (line 1800+)

But on turn 15, the bot is showing readback AGAIN. This suggests:
- Either `_readback_already_shown` was reset to False somewhere, OR
- Tool execution on turn 14 failed silently, leaving `order_created == False`, causing re-entry on turn 15

## Why This is Low Priority
1. Order ultimately commits successfully with all correct data
2. Customer receives confirmation ("Ihre Bestellung wurde aufgenommen")
3. Address extraction is working perfectly now
4. Score is 100% - the loop is a harness detection artifact, not a product failure

## Potential Fixes (if revisited)
1. Add flag `_commitment_attempted` to prevent re-entering readback after first confirmation attempt
2. Make confirmation cycle limit more aggressive (reduce from 2 to 1)
3. Add explicit guard: `if _readback_already_shown and end_call_stage == "idle", skip entire readback block`
